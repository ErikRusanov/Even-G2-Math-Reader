# Plan: Even G2 Math Reader — MVP в 7 независимых итераций

## Context

Цель — довести проект из состояния «research готов, кода нет» до читаемого на очках
teleprompter-MVP для плотной математики. Подробности продукта и ограничений железа — в
`CLAUDE.md` и `docs/` (research, рендер-пайплайн, архитектура).

**Формат этого плана:** 7 итераций (0–6), каждая написана как **самостоятельный бриф** —
рассчитана на запуск с **чистым контекстом**. Поэтому каждая итерация начинается с блока
«Read first», описывает **ожидаемое состояние репозитория на входе** (что сделали прошлые
итерации) и заканчивается чётким критерием «Done». Итерации зависимы по порядку (N опирается
на результат N−1), но запускаются по отдельности.

**Жёсткое правило (из CLAUDE.md):** Итерация 0 — hardware spike — может обнулить дизайн рендера.
Пока она не подтвердила реальный image API на железе, нельзя писать KaTeX-пайплайн вслепую.

**Окружение (проверено 2026-06-20):** Node v25.9.0, npm 11.12.1. Репозиторий чистый: только
`CLAUDE.md`, `README.md`, `docs/`. У пользователя **очки G2 на руках**.

**Общие конвенции для всех итераций:**
- Все вызовы SDK — только через адаптер `src/glasses/` (SDK v0.0.x будет меняться).
- Прямой BLE (`i-soxi/even-g2-protocol` и пр.) — research-only, в коде не используем.
- Математику рендерим **крупно и скроллим**, не ужимаем под страницу.
- Официальные доки SDK: https://hub.evenrealities.com/docs
- В конце каждой итерации — обновить `CLAUDE.md` (Status) и, если узнали факты о железе,
  `docs/01-research-findings.md`.

## Итерация 0 — Spike «hello glasses» (РАЗБЛОКИРУЕТ ВСЁ)

**Read first:** `CLAUDE.md`, `docs/01-research-findings.md` (§3 дисплей, открытые вопросы),
`docs/03-app-architecture.md` (Milestones, Risks).

**Состояние на входе:** чистый репозиторий (только docs).

**Зачем:** вывести одну картинку на очки и снять ответы на 4 блокирующих вопроса — до того,
как написана хоть строка рендера.

**Шаги:**
1. Скаффолд официального шаблона `even-realities/evenhub-templates` (Vite + TS), `npm install`,
   локальный запуск через `evenhub-simulator`.
2. Адаптер `src/glasses/` — тонкая обёртка над `@evenrealities/even_hub_sdk`: `connect()`,
   `sendImage(bitmap)`. Никакой логики, только изоляция SDK.
3. Заготовить 3 статичных PNG **руками/скриптом** (НЕ пайплайном): (a) одна формула крупно,
   (b) та же мелко, (c) шахматка 576×288 для проверки границ surface.
4. Авторизация + QR-sideload через `@evenrealities/evenhub-cli`, пуш на очки.
5. Снять с железа ответы на открытые вопросы:
   - реальный максимальный размер image (≈200×100 или полные 576×288?);
   - картинка обходит 10-строчную teleprompter-структуру или её надо оборачивать в page-пакеты?
   - chunking больших картинок прозрачный или ручной?
   - ловятся ли в кастомном app события **TouchPad / R1**?
6. Записать подтверждённые числа в `docs/01-research-findings.md` (заменить гипотезы фактами).

**Файлы:** `package.json`, `vite.config.*`, `src/glasses/index.ts`, `src/main.ts`,
`public/test/*.png`, апдейт `docs/01-research-findings.md`, `CLAUDE.md`.

**Done:** пользователь видит картинку на очках своими глазами; 4 открытых вопроса в docs
заменены реальными данными.

**Риск:** image API окажется иным — шаг 3 специально щупает границы заранее.

## Итерация 1 — Рендер-пайплайн LaTeX → 4-bit bitmap

**Read first:** `docs/02-math-rendering-pipeline.md` (весь), обновлённый `docs/01` §image API.

**Состояние на входе:** есть скаффолд, `src/glasses/` адаптер, подтверждённый из Итерации 0
реальный размер/формат картинки.

**Зачем:** превратить LaTeX в читаемый на 4-bit зелёном дисплее bitmap; откалибровать кегль и
дизеринг на реальном железе.

**Шаги:**
1. `src/render/` — пайплайн: KaTeX → canvas → даунскейл до 4-bit grayscale → Floyd–Steinberg
   dither → **инверт в white-on-black** (сохраняет тонкие штрихи: дробные черты, Σ, греческие).
2. Параметризовать кегль/толщину под размер, подтверждённый Итерацией 0.
3. Рендерить 2–3 формулы разного размера, пушить через `src/glasses/` на очки.
4. Калибровка читаемости глазами: какой минимальный кегль читается, какой дизеринг лучше.
5. Зафиксировать выбранные параметры рендера в `docs/02`.

**Файлы:** `src/render/katex.ts`, `src/render/dither.ts`, `src/render/index.ts`, апдейт `docs/02`.

**Done:** дробь/сумма/индекс читаются на очках; параметры рендера зафиксированы в docs.

**Риск:** математика нечитаема при доступном размере → fallback: рендерить крупнее и скроллить
(см. Итерацию 3), tiling.

## Итерация 3 — Статичный постраничный рендер на очки

**Read first:** `docs/03-app-architecture.md` (Rendering & caching), `docs/02` (нарезка).

**Состояние на входе:** есть рендер-пайплайн (Итер. 1) и загрузка/выбор файлов (Итер. 2).

**Зачем:** показать целый файл как страницы-картинки с ручным перелистыванием.

**Шаги:**
1. Парсить выбранный `.md` в чередование прозы и формул, рендерить в одну вертикальную «ленту».
2. `src/render/` — нарезка ленты на strips по высоте видимой области очков.
3. Кэш strips по content-hash (`src/cache/`, in-memory).
4. Экран «Teleprompter» в ручном режиме: кнопки вперёд/назад листают strips, пуш через адаптер.

**Файлы:** `src/render/slice.ts`, `src/cache/index.ts`, `src/teleprompter/pages.ts`,
`src/ui/prompter.ts`.

**Done:** реальный файл читается на очках постранично, ручное листание без рассинхрона.

## Итерация 4 — Autoscroll (управление скоростью с телефона)

**Read first:** `docs/03-app-architecture.md` (Autoscroll design), `docs/01` §5 (scroll modes).

**Состояние на входе:** работает статичный постраничный рендер (Итер. 3).

**Зачем:** автоскролл по таймеру с регулировкой скорости — это и есть ядро MVP.

**Шаги:**
1. `src/teleprompter/` — движок скролла: `setInterval`/rAF-аккумулятор продвигает индекс strip,
   пушит картинку на каждом шаге.
2. Регулировка скорости (слайдер на телефоне, напр. 2–30 сек/страница), персист per-file.
3. Pause/resume.
4. Eager pre-render всего файла на входе (прогресс-бар), чтобы скролл не стопорился.

**Файлы:** `src/teleprompter/engine.ts`, `src/teleprompter/speed.ts`, `src/ui/settings.ts`.

**Done:** файл автоскроллится на очках с регулируемой с телефона скоростью, без стопов на
подгрузке. **← это рабочий MVP (Итерации 0→4).**

## Итерация 5 — Управление с очков (TouchPad / R1)

**Read first:** `docs/01-research-findings.md` §5 + ответ Итерации 0 на вопрос о доступе к input.

**Состояние на входе:** работает autoscroll с телефонным управлением (Итер. 4).

**Зачем:** управлять чтением прямо с очков (если Итерация 0 подтвердила доступ к input).

**Шаги (только если Итер. 0 показала, что события доступны):**
1. Подписка на TouchPad/R1 в `src/glasses/`.
2. Маппинг: tap = pause/resume; swipe/long-press = скорость ±.
3. Если input недоступен — задокументировать ограничение, оставить телефонное управление как
   baseline (итерация закрывается выводом, а не кодом).

**Файлы:** `src/glasses/input.ts`, `src/teleprompter/engine.ts` (подключение событий).

**Done:** pause/скорость управляются с очков; либо зафиксировано, что input недоступен.

## Verification (для каждой итерации)

- **Локально:** `npm run dev` + `evenhub-simulator` (превью 576×288 monochrome).
- **На железе:** `evenhub-cli` → QR-sideload → запуск на очках, проверка глазами (читаемость
  математики — главный критерий, симулятор её не заменяет).
- **Критерий приёмки** — блок «Done» соответствующей итерации.
- После итерации обновить `CLAUDE.md` (Status) и `docs/` подтверждёнными фактами о железе.

## Как запускать итерации

Каждую итерацию запускать **с чистым контекстом**, давая фразу вроде:
«Выполни Итерацию N из `.claude/plans/tingly-conjuring-lightning.md`». Агент прочитает блок
«Read first», проверит ожидаемое состояние репозитория и доведёт до «Done».
