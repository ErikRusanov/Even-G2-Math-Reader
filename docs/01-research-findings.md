## TL;DR ‚Äî Is this buildable?

**Yes.** The Even Realities G2 has an **official developer platform (EvenHub)**. Apps are
ordinary **web apps (HTML/CSS/TypeScript)** that run inside the **Even Hub companion phone
app's WebView**. The glasses themselves run no app logic ‚Äî they are a **576√ó288 px
monochrome-green display + input device** (touchpad, gestures, voice, R1 ring).

The **hardest constraint is rendering dense math**. The native text path enforces ~25
characters per line ‚Äî LaTeX-grade fractions, sums, matrices, and subscripts are **impossible
as text**. The only viable path is to **pre-render each formula (or whole page) to a 4-bit
grayscale bitmap** with KaTeX/MathJax and push it as an **image payload**. As of Iteration 0 the
image API is **settled** (see "Open questions ‚Äî RESOLVED" below): send encoded PNG/JPEG bytes per
image container (‚â§288√ó144, up to 4 tiled to fill 576√ó288); the host does the 4-bit conversion.

## 3. Display hardware & limits ‚Äî `confidence: HIGH (specs) / MEDIUM (exact pixel limits)`

- **576√ó288 px, monochrome green, 4-bit grayscale (16 intensity levels).** No RGB.
- **~1200 nits Micro-LED** (notebookcheck).
- **Native teleprompter mode: exactly 10 lines/page, ~25 chars/line, ~7 lines visible at once.**
- **Image containers: each `ImageContainerProperty` is 20‚Äì288 px wide √ó 20‚Äì144 px tall**
  (`confidence: HIGH` ‚Äî from `@evenrealities/even_hub_sdk@0.0.10` type definitions, PB
  `Width 20~288 / Height 20~144`). The rumored "~200√ó100" cap was just the official
  `image` template's *chosen* size, **not** a hard limit ‚Äî **REFUTED as a limit**.
- **The full 576√ó288 surface IS addressable.** A single image can't span it (‚â§288√ó144), but a
  page may hold **up to 4 image containers** (`imageObject` max_count 4) + up to 8 text + 12
  total, so a **2√ó2 tile of 288√ó144 images covers the entire 576√ó288 surface**. Text/event
  containers can already be full-surface (576√ó288). ‚úÖ **CONFIRMED eyes-on-glass (2026-06-20):**
  the 4-tile checkerboard rendered full-surface. ‚öÝÔ∏è **BUT very slow** ‚Äî pushing 4 tiles
  (4√ó serial `updateImageRawData` over BLE) took many seconds. **Implication for scroll:** do
  **not** repaint the full surface per frame; push a single image region per step (or only the
  changed tile). This is a binding constraint on Iterations 3‚Äì4.

### ¬ßimage-perf ‚Äî why image pushes are slow, and what to minimize ‚Äî `confidence: HIGH`

Researched 2026-06-20 (the Iteration-5 HUD "responds terribly" investigation). The on-glass
slowness has a clear root cause and a clear lever:

- **Cost = container PIXEL AREA + a fixed per-push round-trip. NOT our PNG/JPEG byte size.** The
  Even Hub host **re-encodes each container to a fixed-size bitmap (resize + grayscale) before the
  BLE hop**, so a smaller/cleverer PNG buys nothing ‚Äî the bytes that actually cross BLE are fixed by
  the container's dimensions. Smaller container = fewer bytes = faster; clever compression = no
  change.
- **Field-measured rates:** a 50√ó50 image ‚âà **4 FPS (~250 ms)**, 30√ó30 ‚âà **9 FPS (~107 ms)** (the
  fixed round-trip dominates at small sizes); a max-size **576√ó136 ‚âà 3 s** (area dominates at large
  sizes). Extrapolating, a **288√ó144 tile ‚âà ~1.5 s**, a full 4-tile 576√ó288 page ‚âà **~3 s+**.
- **Pushes are serial** (SDK enforces queue mode ‚Äî `updateImageRawData` must await the previous), so
  **N tiles ‚âà N √ó (area cost) + N √ó round-trip**, and you visibly see them paint one-by-one.
- **Two levers, in order:** (1) **repaint fewer tiles** ‚Äî touch only the one(s) that changed; (2)
  **repaint a smaller container** where the layout allows it. There is **no slot for a small
  dedicated overlay** during reading: a full-width page taller than 144 px needs all 4 containers
  (SDK cap), so transient UI (e.g. the speed HUD) must reuse an existing tile, and the cheapest such
  update is **a single tile, pushed atomically** (Iteration 5 HUD).
- Sources: even-realities/EvenDemoApp#36 (~3 s for a max image); zenn.dev/bigdra eveng2-sdk-features
  (50√ó50‚Üí4 FPS / 30√ó30‚Üí9 FPS, "host handles grayscale", "queue mode, no concurrent send");
  hub.evenrealities.com docs.

**A second, content-dependent bottleneck ‚Äî bridge serialization (partially understood; base64
workaround REFUTED on hardware 2026-06-20):** before the BLE hop, the image payload crosses the
**WebView‚Üînative bridge**, and *page loads* balloon in a way pure BLE can't explain ‚Äî observed
bilet-01 slide 1 ‚âà 11 s vs the **denser** slide 2 ‚âà 30 s. Area-fixed BLE would be ~constant across
slides, so the extra cost scales with payload/content. Mechanism (from the `.d.ts`):
`ImageRawDataUpdate.imageData` is `number[] | string | Uint8Array | ArrayBuffer`, and `toJson`
**converts a Uint8Array into a `number[]`** (`List<int>`) that is JSON-serialized to the host ‚Äî a
~40 KB tile ‚Üí a **~40 000-element array**, and a heavily-dithered (high-entropy) page ‚Üí bigger PNG ‚Üí
bigger array ‚Üí slower.
- **Tried:** pass a **base64 string** instead (the `.d.ts` says *"‰πüÂÖÅËÆ∏‰ºÝ base64 stringÔºàÁî±ÂÆø‰∏ªËá™Ë°åÂ§ÑÁêÜÔºâ"*).
  ‚úÖ Works in the **simulator**. ‚ùå **On real G2 firmware the slides silently DON'T render** ‚Äî you're
  left looking at the native menu text. The device host does not decode the base64 the way the JS
  simulator host does. **REVERTED** to the Uint8Array path (the only hardware-safe one).
- **Implication / still-open:** the byte‚Üí`number[]` bridge cost is real but we can't dodge it by
  changing the payload TYPE. The remaining levers are payload **size/entropy** (lighter dithering or
  a cleaner image the host quantizes itself) and **pixel count** (smaller pages / fewer tiles) ‚Äî both
  change output quality or UX, so they need eyes-on-glass before committing. Whether the 11‚Üí30 s is
  truly the bridge `number[]` build vs. the host's per-tile PNG-decode+gray4 (also content-dependent)
  is **not yet isolated** ‚Äî needs on-device timing.

**MEASURED ON-DEVICE (2026-06-20) ‚Äî root cause settled.** Added a diagnostic that times the
`Uint8Array‚Üínumber[]` build (`ser`) separately from the full `updateImageRawData` round-trip
(`total`), per tile, with the payload size. Real G2, bilet-01, manual paging:
- `ser` = **0‚Äì7 ms** ‚Äî the bridge serialization is **negligible**. The base64 theory above was a red
  herring; `number[]` build is not the cost.
- `net` (= total ‚àí ser ‚âà host decode + gray4 + BLE) is the **entire** cost, and is **uncorrelated
  with payload size**: a 6.4 KB tile took 11 s while a 12.4 KB tile took 5.5 s. So smaller/cleaner
  PNGs would **not** help ‚Äî payload-size optimization is dead.
- Per-tile cost is a **fixed ~3 s when fresh** (page 1: 4 tiles ‚âà 12.6 s) that **degrades to ~7‚Äì13 s
  under rapid paging** (page totals climbed 12.6 ‚Üí 33 ‚Üí 40 s, then settled ~29 s) ‚Äî i.e. a per-push
  floor plus congestion/host-memory growth. A 288√ó144 tile costs about the same ~3 s as the
  EvenDemoApp's full 576√ó136, confirming the cost is **per-image-update overhead, not per-pixel/byte**.
- **Conclusion:** the only effective lever is **fewer image pushes per page**. The minimum to cover
  576√ó288 is 4 containers, so a full-surface page is ~12 s floor with no payload escape. **Chosen fix
  (Iteration 6): 2-tile reading pages** ‚Äî render the document into 576√ó144 pages (`layoutTile1x2`,
  top half), 2 pushes/page ‚âà ~2√ó faster, at the cost of a shorter reading window + more page flips.
  Diagnostic (`src/glasses/perf.ts` + `prompter` readout) stays in until the 2√ó is confirmed
  eyes-on-glass, then gets removed.
Sources: `@evenrealities/even_hub_sdk@0.0.10` `index.d.ts` (`ImageRawDataUpdate`, `toJson` /
`normalizeImageData`).

## 4. BLE protocol ‚Äî `confidence: HIGH for G1, MEDIUM for G2`

Relevant only if you bypass the official SDK and talk to the glasses directly (not recommended
for v1, but useful to understand the substrate):

- **Nordic UART Service (NUS) UUIDs** (confirmed across multiple primary sources):
  - Service: `6E400001-B5A3-F393-E0A9-E50E24DCCA9E`
  - Write (TX): `6E400002-B5A3-F393-E0A9-E50E24DCCA9E`
  - Read/Notify (RX): `6E400003-B5A3-F393-E0A9-E50E24DCCA9E`
- **G1 uses two BLE radios ‚Äî one per arm** ‚Äî each a separate UART stream; sends are sequential
  (left, then right).
- **G2 teleprompter pages** transmit as **10-line UTF-8 chunks with `\n` separators**, page
  numbers as **0-indexed varints** (binary-level analysis in `i-soxi/even-g2-protocol`).
- ‚öÝÔ∏è All G2 BLE detail is **reverse-engineered**, not official, and may not match current
  firmware. Payload/MTU specifics (180/194/204-byte packet claims) were **refuted/unconfirmed** ‚Äî
  do not rely on them.

## 5. Autoscroll / scroll speed ‚Äî `confidence: MEDIUM`

The **native** Even teleprompter offers **three scroll modes** (primary source,
evenrealities.com/teleprompter-glasses):

1. **AI-paced** ‚Äî adapts to the speaker's speech rate via voice recognition.
2. **Fixed speed** ‚Äî a consistent WPM setting.
3. **Manual** ‚Äî via the **Even R1 smart ring** or **tapping the G2 TouchPad**.

> ‚úÖ **CONFIRMED (Iteration 0, SDK types):** custom web apps **do** receive glasses input.
> `bridge.onEvenHubEvent` delivers `OsEventTypeList` gestures ‚Äî
> `CLICK_EVENT`(0, tap), `DOUBLE_CLICK_EVENT`(3), `SCROLL_TOP_EVENT`(1), `SCROLL_BOTTOM_EVENT`(2)
> ‚Äî and `Sys_ItemEvent.eventSource` (`EventSourceType`) distinguishes the source:
> **right touchpad** (`TOUCH_EVENT_FROM_GLASSES_R`), **left touchpad** (`..._GLASSES_L`), and the
> **R1 ring** (`TOUCH_EVENT_FROM_RING`). So on-glasses speed control is viable.
> Caveats: (a) input is captured by a **text/list container with `isEventCapture: 1`** ‚Äî image
> containers can't capture, so a full-surface text layer must sit behind the images;
> (b) **taps/double-taps/lifecycle arrive via `sysEvent`, scrolls via `textEvent`** (scroll events
> carry no `eventSource`); (c) `CLICK_EVENT === 0` and protobuf omits zero-valued fields, so a tap
> arrives as a `sysEvent` with `eventType === undefined` ‚Äî coalesce before comparing.
> Still **pending eyes-on-glass**: confirming the physical gesture‚Üíevent mapping (e.g. which
> swipe direction is SCROLL_TOP, whether the ring actually emits these) on real hardware.
> Fallback remains: drive autoscroll from a `setInterval` and set speed from the phone UI.
>
> ‚úÖ **WIRED (Iteration 5):** the reader now subscribes to these gestures via the adapter's
> `onInput` and maps them in `src/teleprompter/gestures.ts`: **tap = play/pause**, **scrollUp =
> faster** (‚àídwell), **scrollDown = slower** (+dwell), **doubleTap / system-exit = leave reader**.
> Speed swipes step multiplicatively (~25%/swipe via `stepSpeed`) and sync the engine + phone slider
> + per-file persistence + a brief on-glass status flash. The chosen direction (swipe-up = faster)
> is a documented assumption isolated to one `switch` in `gestures.ts` ‚Äî **flip those two cases if
> eyes-on-glass shows the physical direction inverted**. The ring actually emitting events and the
> swipe‚ÜíSCROLL_TOP/BOTTOM direction are the only items still needing hardware confirmation.

## 6. Math rendering ‚Äî `confidence: MEDIUM (engineering inference) / HIGH for legibility (eyes-on-glass)`

- **No existing G2 math/LaTeX project was found.** This is novel territory.
- LaTeX-grade math **cannot** be rendered as text (25 char/line limit).
- **Pipeline (built in Iteration 1):** LaTeX ‚Üí **MathJax** (SVG path output) ‚Üí canvas ‚Üí downscale
  to **4-bit grayscale** ‚Üí send as image. Use **Floyd‚ÄìSteinberg dithering** and **invert to
  white-on-black** (MathJax renders dark glyphs) to preserve thin strokes (fraction bars, Œ£, Greek)
  at low bit depth. **MathJax was chosen over KaTeX:** KaTeX's HTML output only rasterizes via SVG
  `<foreignObject>`, which **taints the canvas on WebKit/WKWebView** and breaks the `getImageData`
  read-back the dither needs (and `toBlob` export); MathJax SVG paths never taint. See `docs/02`.
- ‚úÖ **CONFIRMED eyes-on-glass (Iteration 0, 2026-06-20):** dense math (fraction bar + Œ£ with
  limits + sub/superscripts) is **perfectly legible at 4-bit**. The **ideal glyph scale is the
  `formula-small` probe** ‚Äî a formula drawn into a **~220√ó80 container** (serif, base font
  ‚âà 27 px in that probe). The **`formula-large` (288√ó144) probe was too big** ‚Äî unnecessarily
  large. ‚úÖ **Iteration 1 calibrated & confirmed eyes-on-glass (2026-06-20): `pxPerEx = 8`** (ideal)
  ‚Äî every sample reads cleanly and tightly; the reference series formula ‚Üí ~145√ó51 px, well under
  `formula-small`, leaving generous vertical room for scroll. (Probes generated by
  `scripts/gen-test-images.mjs`.)
- Mature LaTeX‚Üíimage references: `DMOJ/texoid`, `mneri/pnglatex`, `klatexformula`.

## Open questions ‚Äî RESOLVED in Iteration 0 (2026-06-20)

Answered from `@evenrealities/even_hub_sdk@0.0.10` type definitions + the official
`even-realities/evenhub-templates` **`image`** template (authoritative for the API), and
**confirmed eyes-on-glass on real G2 hardware on 2026-06-20** by running the spike (`src/main.ts`).

1. **Input access ‚Äî ‚úÖ YES.** `bridge.onEvenHubEvent` exposes tap / double-tap / scroll from the
   **glasses touchpads (L/R)** and the **R1 ring**, with `eventSource` identifying which. Requires a
   text container with `isEventCapture: 1` to capture; taps‚Üí`sysEvent`, scrolls‚Üí`textEvent`;
   `CLICK_EVENT===0` is omitted on the wire. *Eyes-on-glass:* confirm gesture‚Üíevent direction & that
   the ring emits events. (See ¬ß5.)
2. **Image API ‚Äî ‚úÖ Send ENCODED bytes; it bypasses the teleprompter.** `bridge.updateImageRawData`
   takes a `Uint8Array` of **encoded PNG/JPEG bytes** for a given `ImageContainerProperty`; the host
   **decodes, resizes to the container, and converts to 4-bit grayscale itself** (returns
   `imageToGray4Failed` if conversion fails). We do **not** pre-pack 4-bit pixels, and images are
   their own containers ‚Äî **not** wrapped in the 10-line teleprompter page structure.
3. **Payload/MTU ‚Äî ‚úÖ Transparent.** `updateImageRawData` accepts the whole `imageData` in one call;
   the SDK's fragmentation fields (`mapSessionId/mapTotalSize/mapFragmentIndex/...`) are marked
   *reserved/unused* ‚Äî the host chunks internally. **Constraint:** image updates must be **serial**
   (one in flight at a time) ‚Äî the adapter enforces this with a promise queue.
4. **Full-surface mode ‚Äî ‚úÖ YES, via tiling, but SLOW.** A single image container is ‚â§288√ó144, but a
   page can hold **up to 4 image containers**, so a **2√ó2 grid of 288√ó144 tiles covers the full
   576√ó288** ‚Äî confirmed eyes-on-glass (Probe 3). ‚öÝÔ∏è **Pushing 4 tiles is very slow** (4√ó serial
   `updateImageRawData` over BLE, many seconds). **Do not repaint the full surface per scroll
   frame** ‚Äî push one image region per step, or update only the changed tile. (See ¬ß3.)

5. **Math legibility & target size ‚Äî ‚úÖ CONFIRMED.** Dense math reads perfectly at 4-bit. **Ideal
   scale = the `formula-small` probe (~220√ó80 container)**; the `formula-large` (288√ó144) probe is
   too big. Iteration 1 calibrates KaTeX output to the `formula-small` scale. (See ¬ß6.)

### Iteration-0 spike (this repo) ‚Äî how it was run on hardware
- `npm install && npm run gen:test-images` ‚Üí builds 3 probe images into `public/test/`.
- Local preview: `npm run dev` (Vite on :5173), optionally `npm run simulate` (`evenhub-simulator`).
- **On real glasses (the flow that worked):**
  1. Enable **Developer Mode**: sign in at **hub.evenrealities.com/login**, then **force-quit** the
     Even Realities phone app and reopen ‚Üí an **Even Hub** tab (top-right) appears.
  2. `npm run dev`, then `npx evenhub qr --url http://<LAN-IP>:5173` (use the **current** `en0` IP ‚Äî
     it's DHCP and changes; `ipconfig getifaddr en0`).
  3. In the app: **Even Hub ‚Üí Scan QR** ‚Üí point at the terminal QR. App loads in the phone WebView,
     glasses render within ~1s. **No auth token needed** for local QR testing. (`evenhub pack` is only
     for packaged distribution.)
- ‚öÝÔ∏è Don't confuse with **Even Terminal** (`@evenrealities/even-terminal`) ‚Äî a *separate* product
  that mirrors a laptop AI agent (Claude Code/Codex) onto the glasses and asks for an auth token.
  That is **not** our path.
- The app cycles 3 probes (**tap**=next, **double-tap**=exit) and logs every gesture + send-result on
  the phone panel and the glasses status line ‚Äî that's how the *eyes-on-glass* answers were read.

## Caveats

- G2 BLE/protocol facts are reverse-engineered, not official, and firmware-version-sensitive.
- The math-via-image pipeline is an **inference from confirmed hardware constraints** ‚Äî no
  existing project validates it empirically. Build a hardware spike early.
- SDK v0.0.10 is early; image-size limits and display-mode APIs may change.
- Several precise numbers (exact packet sizes, 576√ó136 BMP, 2000-char text path) were
  **refuted or split** in verification ‚Äî see [`04-sources.md`](./04-sources.md) refuted list.
